## Findings → Phase 1b backlog

### Finding #1: Sanitizer false-positives on legitimate strings

`flyn-sanitize` flags:
- `--dangerously-skip-permissions` in `backends/claude_p.py:31` — REQUIRED for headless `claude -p`
- `https://api.telegram.org/bot{token}/sendMessage` in `adapters/channels/telegram.py:88` — REQUIRED for outbound Telegram

**Fix:** Add a `.sanitize-allowlist` file format that lets specific files/lines opt out of specific pattern checks with a justification comment. Phase 1b criterion.

### Finding #2: Worktree allocation not idempotent under stale state

When prior tasks leave orphaned worktree registrations or branches, `WorktreeManager.allocate()` fails with `CalledProcessError` at the second `git worktree add`. Reproduced live during verification (task T-0002 failed at `decomposed → failed` immediately).

**Fix:** `WorktreeManager.allocate()` should:
1. `git worktree prune` first to clean stale registrations
2. `git branch -D <branch>` if the branch exists but has no live worktree
3. THEN attempt `git worktree add`

Phase 1b criterion. Also captured in `KNOWLEDGE/16-...`.

### Finding #3: 0-byte capture files don't block state advancement

If `claude -p` exits early (e.g., missing `--verbose` as we hit during the first overnight e2e), the capture file is empty but `exit_code=0` and the task happily advances to `deliverable_ready`. The reviewer then "approves" the empty diff.

**Fix:** Dispatcher should:
- Check `capture_path.stat().st_size < 100` (no stream-json events ≈ no work)
- Refuse to advance, mark task `failed` with diagnostic

Reviewer should:
- Treat empty diff as `passed=false, severity=critical, area=correctness, note="builder produced no diff"`

Phase 1b criterion. Already documented in `KNOWLEDGE/15-claude-p-stream-json-needs-verbose.md`.
