§ I · PROBLEM

The gap between
conversation and memory.

Flyn handles channels well. It does not yet remember them.

The State Today

A Telegram conversation with Beth on Monday does not inform Flyn's behaviour on Wednesday.

The OpenClaw built-in memory captures agent turns, not raw inbound traffic. Graphiti captures facts, not the message stream that produced them. The memory-router has ten read sources — none of them include actual conversation history.

The Promise

Every inbound message becomes searchable conversation memory.

Security-grade encryption for replay. Entity-graph promotion for cross-channel "when did X happen" queries. A small, reversible slice that proves the architecture before fanning out to WhatsApp, iMessage, and email.

One service, six tiers, one new hook.

§ III · DATA FLOW

From key-press to recall, annotated.

A single Telegram message's lifecycle through the system, with timing budgets.

t = 0 ms
User · Telegram client
Sends a message in their chat with @flyn_4c_bot.
~100 ms
OpenClaw Gateway · existing
Picks up the message via getUpdates polling.
~110 ms
flyn-conv-memory-tap · new internal hook
Hook fires before openclaw's agent sees the message. Builds a POST payload with source, body, raw_payload, and a stable dedup_key.
~115 ms
server.py · /api/memory/ingest
Routes by event_type: conversation_message dispatches to the new conv_write adapter.
~118 ms
dedup · existing
Check (source, dedup_key). Drops Telegram resends transparently.
~120 ms
redact.py · existing
Strip secrets from body and raw_payload using the existing allowlist.
~122 ms
owner.resolve_from_chat
Map ("telegram", sender_id) → Owner("ryan") via principals.json. Unknown senders fail loudly — never silently filed.
~125 ms
encrypted_raw.seal
AES-GCM seal of the redacted raw payload with the per-owner key fetched from macOS Keychain. Subsequent calls hit the cached key.
~130 ms
ConvDb("ryan").write
Insert into messages with summary = NULL. FTS5 trigger indexes the redacted body. Returns row_id.
~135 ms
disk queue
Enqueue a summarise job. Reuses the existing memory-router disk queue with a new conv-summarize/ prefix.
~140 ms
httpx · POST Graphiti
Fire-and-forget episode with group_id="flyn-ryan". The entity extractor runs server-side. Failure logs but does not block.
~150 ms
Return
200 OK bubbles back to the hook, which returns to openclaw. Openclaw continues its normal agent processing untouched.
Async
Background worker · summarizer thread
Within 3–8 seconds, a worker pulls the job, calls Ollama, and writes the summary back. The FTS5 trigger updates the summary index. Job file is deleted.
§ IV · DECISIONS

Eleven locked, in order of stakes.

Every architectural choice that constrains the rest of the design.

01SCOPE

Extend memory-router

Conversation lives inside the existing service as a new tier. No new daemon, no new port, no new launchd unit.

02SLICE

Telegram first

Prove the architecture on one channel. WA, iMessage, email are separate small hooks added later atop the same conv tier.

03TENANCY

Per-owner physical isolation

Each owner gets ~/.flyn/memory-router/conv/<owner>.db. Cross-owner reads require an explicit grant; audit-logged.

04SCHEMA

Raw + parsed + summary

Three columns per message. Encrypted raw for replay, redacted body for FTS5, Gemma summary for token-efficient retrieval.

05CADENCE

Async summariser

Ingest returns 200 immediately with summary = NULL. A background worker fills the summary within ~5 s.

06GRAPH

Every message → Graphiti

Entity extraction is free if we promote each message as an episode. Cross-channel "when did X happen" becomes a graph query.

07SECRECY

AES-GCM encrypted raw

Per-owner key in macOS Keychain. Raw is only decryptable via flyn-mem conv replay — no REST endpoint surfaces it.

08RETRIEVAL

conv_read · 11th adapter

Plugs into the existing query fan-out. RRF merges it with the other ten sources for cross-system answers.

09INGRESS

OpenClaw internal hook

flyn-conv-memory-tap fires on every inbound Telegram message and POSTs to :8400/api/memory/ingest.

10STRUCTURE

Dedicated conv/ module

Storage, owner, encryption, summariser each in their own file. ≤ 200 lines apiece. Easy to retire later.

11QUALITY

18 tests + live smoke

Security-sensitive code gets full TDD. Everything else verified by a real Telegram message to @flyn_4c_bot.

§ V · COMPONENTS

Six new modules, each with one clear job.

All under flyn_memory_router/conv/ or adapters/. Each ≤ 200 lines.

≤ 200 LOC
CONV / 01

schema.py

SQLite tables + FTS5 + triggers for a single owner's conversation database.

  • messages (id, ts, body, summary, encrypted_raw)
  • messages_fts (body, summary)
  • idx on thread_id, sender_id
  • WAL · idempotent migrations
≤ 150 LOC
CONV / 02

owner.py

Registry, grants, audit log. The chokepoint for who-sees-what.

  • resolve_from_chat → Owner
  • viewer_can_read · default-deny
  • grant / revoke (audit-logged)
  • list_accessible_owners
≤ 150 LOC
CONV / 03

encrypted_raw.py

Per-owner AES-GCM. Keys live in macOS Keychain, never on disk.

  • seal / unseal · 12-byte nonce
  • KeychainLocked · explicit failure
  • Tamper detection via auth tag
  • No REST surface — CLI only
≤ 150 LOC
CONV / 04

summarizer.py

Worker thread polling a disk queue. Calls Ollama, writes summary back.

  • gemma4:e4b · 30 s timeout
  • 1 s busy poll · 10 s idle poll
  • Backlog visible via flyn-mem conv health
  • Daily backfill pulse
≤ 200 LOC
ADAPTER / 01

conv_write.py

The five-step write sequence. Implements existing MemoryAdapter protocol.

  • owner.resolve → encrypted_raw.seal
  • ConvDb.write → enqueue → Graphiti POST
  • All steps fail-open as WriteResult
  • Never raises to the ingest pipeline
≤ 200 LOC
ADAPTER / 02

conv_read.py

11th adapter in the existing query fan-out. FTS5 over body + summary.

  • Resolves viewer from $USER
  • Queries each accessible owner's db
  • Cross-owner reads write audit
  • Default-included in flyn-mem query

§ VIII · PERFORMANCE & FAILURE

Budgets and how it breaks.

Every path has a target. Every failure has a behaviour. Nothing crashes silently.

TARGET
NOTES
Hook → ingest → 200 OK
p95 < 250 ms
Section III walks ~150 ms. Headroom for Keychain cold-start on first message of the day.
Summariser roundtrip
3–8 s typical
Asynchronous; never blocks the hook. Gemma cold-start adds a few seconds on the first message after idle eviction.
conv_read.query
p95 < 20 ms
FTS5 against local SQLite. Joined with the existing memory-router query budget.
End-to-end query (all 11)
p95 < 500 ms
Same cap as the existing /api/memory/query surface. The conv adapter is one of the cheapest.
Owner unresolvable
WriteResult(ok=false). Message not stored — no orphan owner data. Logged to conv-writes; counter exposed.
Keychain locked
Hook still acks 200 to openclaw. Message not stored unencrypted. Counter increments. Recovery: unlock Mac, resend.
Summariser timeout
Row keeps summary=NULL. Visible in conv health backlog. Auto-retried by the daily backfill pulse.
Graphiti POST fails
conv.db write still succeeded. Logged. Backfilled later by a lint job. Local retrieval unaffected.
FTS5 corruption
Read returns []. Source-error logged. Rebuildable via flyn-mem conv rebuild.
Disk full / 500 MB / owner
Health pulse warns. No auto-delete in slice 1. flyn-mem conv archive --before arrives in a future slice.
